# AI 规划与审核执行手册（Planner + Reviewer）

更新时间：2026-02-17
适用仓库：`/Users/xuyicheng/杀戮游戏复刻2.0`

## 1. 目标

这份手册用于固定“你提需求 -> AI 工程师实现 -> 审核通过后提交”的协作流程，减少返工与跑偏。

## 2. 角色分工

1. 你（产品/机制负责人）
   - 定目标、定范围、定取舍。
   - 最终批准是否合入。
2. AI 工程师（Implementer）
   - 严格按任务卡与白名单改动。
   - 提供可复验证据，不做范围外重构。
3. 我（Planner + Reviewer）
   - 先产出计划与提示词。
   - 实现后按严重级别审核，给出通过/驳回结论。

## 3. 标准工作流（RPI）

## 3.1 Research（研究）

要求：
1. 只调研，不改代码。
2. 输出“文件 + 行号 + 当前行为 + 风险”。
3. 结论必须可追溯到代码锚点。

交付物：
1. `docs/tasks/<task-id>/plan.md` 的“现状与风险”段。

## 3.2 Plan（规划）

要求：
1. 拆成可执行步骤，步骤粒度到文件级。
2. 明确“包含/不包含”边界。
3. 列出改动白名单与验证命令。

交付物：
1. `docs/tasks/<task-id>/plan.md`
2. 白名单文件列表（workflow-check 依赖）

## 3.3 Implement（实施）

要求：
1. 严格按计划执行，不扩 scope。
2. 每完成一步就运行最小验证。
3. 写 `verification.md` 记录命令与结果。

交付物：
1. 代码改动（仅白名单内）
2. `docs/tasks/<task-id>/verification.md`
3. `docs/tasks/<task-id>/handoff.md`

## 4. 上下文工程规则（Vibe Coding 复杂工程版）

1. 主会话只保留“当前任务必需上下文”。
2. 搜索与定位尽量隔离（子会话/子代理），主会话只接收结论与行号。
3. 每 20-30 分钟做一次压缩：
   - 当前目标
   - 关键文件锚点
   - 已完成与下一步
4. 上下文过长时必须新开会话，以上一轮压缩文档继续，不硬扛长对话。

## 5. 审核分级与结论规则

## 5.1 严重级别

1. `S0 阻断`
   - 机制错误、存档破坏、流程中断、越界改动、门禁失败。
   - 结论：直接驳回，不允许提交。
2. `S1 高风险`
   - 行为与计划不一致、回归风险高、缺核心验证。
   - 结论：要求修复后复审。
3. `S2 一般问题`
   - 文档缺漏、可读性问题、非关键警告。
   - 结论：可带条件通过或要求补齐后通过。

## 5.2 审核通过门槛

必须同时满足：
1. `make workflow-check TASK_ID=<task-id>` 通过。
2. 改动全部在白名单内。
3. `verification.md` 有可复验步骤和结果。
4. 关键链路至少 1 条主路径 + 1 条边界用例。
5. 无 S0/S1 未解决问题。

## 6. 提交流程

1. 我先给出审核结论：`通过` / `驳回`。
2. 只有“通过”时才允许提交。
3. 提交信息格式遵循：
   - `feat(module): ...（task-id）`
   - `fix(module): ...（task-id）`
   - `chore(scope): ...（task-id）`

## 7. 快速执行清单

每次任务开工前：
1. 创建任务：`make new-task TASK_ID=<task-id>`
2. 先写计划：`docs/tasks/<task-id>/plan.md`
3. 开发前确认白名单
4. 开发后更新 `verification.md` 和 `handoff.md`
5. 运行：`make workflow-check TASK_ID=<task-id>`
6. 进入审核与提交阶段
