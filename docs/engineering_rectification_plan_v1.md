# 工程整改规划 V1（模块化落地）

更新日期：2026-02-16  
适用项目：`/Users/xuyicheng/杀戮游戏复刻2.0`

## 1. 这份规划的目的（先讲人话）

这份规划要解决四件事：

1. 保住当前可玩版本，不因为重构把项目改挂。
2. 把业务逻辑从场景/UI脚本里逐步搬到模块层。
3. 每一步都可验收、可回滚，避免一次性大改。
4. 最终形成“规则模块化、UI壳层化、数据驱动、可测试”的工程结构。

---

## 2. 改造方向（专业术语）

目标架构：轻量 `Clean Architecture` + 分层解耦。

分层职责：

1. `UI Layer`：只展示与输入，不承载规则。
2. `Application Service Layer`：流程编排、命令执行、状态写入入口。
3. `Domain Layer`：规则、状态机、效果结算、业务约束。
4. `Infrastructure Layer`：存档、内容导入、日志、随机数。

关键改造模式：

1. `Command/Query Separation (CQS)`：写操作和读操作分离。
2. `Domain Service`：跨对象规则集中到服务，不散落在场景脚本。
3. `DTO/ViewModel`：UI读取投影模型，不直接操作领域对象。
4. `Versioned Save Schema`：存档版本化与迁移。
5. `Deterministic RNG`：固定种子与随机流可复现。

## 3. 分阶段路线图（建议 8 周）

| 阶段 | 周期 | 人话目标 | 专业方向 | 里程碑 |
|---|---:|---|---|---|
| Phase 0 基线封板 | 0.5周 | 先保住当前可运行版本 | Baseline Freeze + ADR | 基线分支可稳定运行 |
| Phase 1 架构收口 | 1周 | 把目录和职责讲清楚 | Boundary Clarification + Contract First | 模块边界与接口草案完成 |
| Phase 2 场景去业务化 | 1.5周 | 把商店/事件/营火/奖励的写状态逻辑搬走 | Scene Orchestration Refactor | 场景只发指令，不直接写核心状态 |
| Phase 3 战斗核心内聚 | 1.5周 | 战斗规则统一管理 | Battle Domain Consolidation | 战斗逻辑由 battle/effect/buff 统一驱动 |
| Phase 4 存档与复现拆分 | 1周 | 存档不再一个巨文件 | Serializer Segmentation + Replay | 存档分拆并支持版本迁移 |
| Phase 5 内容管线统一 | 1周 | 内容来源统一，不再双轨 | Single Source of Truth | 奖励/商店/事件统一读管线产物 |
| Phase 6 UI壳层化 | 1周 | UI只显示不算规则 | UI Shell + ViewModel | UI层不直接改领域状态 |
| Phase 7 测试与门禁 | 0.5周 | 防止回归和反屎山 | Quality Gate + Regression Tests | 合并前门禁固定执行 |

## 4. 阶段细化计划

### Phase 0：基线封板

目标：

1. 冻结当前可玩基线，作为重构回退点。
2. 记录当前最小主流程验收路径。

交付物：

1. 基线标签与变更说明。
2. 最小流程验收文档（地图 -> 战斗 -> 奖励 -> 回地图）。

验收标准：

1. 基线版本可稳定启动并跑通主流程。
2. 后续任务都能以基线对比回滚。

---

### Phase 1：架构收口

目标：

1. 明确模块边界、输入输出、允许依赖方向。
2. 统一命名，消除语义重复模块。

关键动作：

1. 明确 `persistence` 与 `seed_replay` 的最终关系（建议合并为一个主模块）。
2. 给每个模块补充 `Contract`（输入、输出、状态变更责任）。
3. 清理空壳目录和“临时占位”目录命名。

交付物：

1. 更新后的模块架构文档。
2. 每模块接口草案（文档级）。

验收标准：

1. 模块边界文档可直接指导任务派发。
2. 不再出现“场景直接跨模块写状态”的新代码。

---

### Phase 2：场景去业务化

目标：

1. 场景脚本只负责交互和展示。
2. 所有写状态逻辑进入应用服务层。

关键动作：

1. 新增服务：`run_flow_service`、`reward_service`、`shop_service`、`event_service`、`rest_service`。
2. 把 `RunState` 写操作从 `scenes/*` 挪到服务层命令函数。
3. 场景只接收服务返回结果并渲染。

交付物：

1. 服务层代码与调用接线。
2. 场景脚本瘦身后的对照清单。

验收标准：

1. `scenes/shop`、`scenes/events`、`scenes/map/rest` 不直接写金币/牌组/楼层。
2. 主流程行为保持一致。

---

### Phase 3：战斗核心内聚

目标：

1. 战斗规则集中，不再散在 UI、Enemy、Card 各处。
2. 状态机、效果栈、Buff触发链闭环。

关键动作：

1. 引入 `BattleContext/BattleState` 作为战斗态统一入口。
2. 弱化全局静态单例，改为按战斗实例注入服务。
3. 统一伤害来源、触发时机、阶段迁移约束。

交付物：

1. 战斗领域服务层。
2. 阶段机与效果链路回归用例。

验收标准：

1. `DRAW -> ACTION -> ENEMY -> RESOLVE` 可稳定循环。
2. 多段伤害与状态衰减结果一致可复现。

---

### Phase 4：存档与复现拆分

目标：

1. 拆分超大 `save_service`，降低复杂度。
2. 保证存档兼容和随机可复现。

关键动作：

1. 序列化按域拆分：`run_meta`、`map`、`inventory`、`card`。
2. 新增存档版本迁移器。
3. 固化复盘日志结构（seed/floor/node/action）。

交付物：

1. 分拆后的 serializer 结构。
2. 存档迁移策略说明。

验收标准：

1. 存档往返测试通过。
2. 同 seed 在同版本下关键流程一致。

---

### Phase 5：内容管线统一

目标：

1. 消除“手写资源 + 生成资源”双轨并存。
2. 让奖励池/商店池/事件加牌统一走内容数据。

关键动作：

1. 奖励模块改为读取 `content_pipeline` 产物。
2. 扩展导入能力到敌人/遗物/事件。
3. 标准化导入错误报告。

交付物：

1. 统一内容源配置。
2. 导入脚本扩展计划与执行结果。

验收标准：

1. 新增卡牌只需改数据并导入即可生效。
2. 奖励、商店、事件不再硬编码卡池。

---

### Phase 6：UI壳层化

目标：

1. UI只关心展示模型，不改业务状态。
2. 替换视觉资源不影响规则层。

关键动作：

1. 引入 ViewModel 适配层。
2. 清理 UI 层中规则判断和状态写入逻辑。
3. 统一 HUD/状态/奖励显示组件接口。

交付物：

1. UI读取模型规范。
2. 页面级 ViewModel 映射代码。

验收标准：

1. `scenes/ui` 不包含核心规则写入。
2. UI资源替换不影响玩法行为。

---

### Phase 7：测试与门禁固化

目标：

1. 防止回归和“屎山反弹”。
2. 把工程规范变成自动门禁。

关键动作：

1. 建立最小自动化测试集。
2. 合并前固定执行 workflow-check + 模块测试。
3. 跨模块改动强制更新契约文档。

最小测试集合：

1. 战斗阶段机迁移测试。
2. 地图推进一致性测试。
3. 存档往返一致性测试。
4. 奖励写回状态测试。

验收标准：

1. 所有关键测试通过才允许合并。
2. 每个任务都有可复验记录。

## 6. 防屎山硬规则（执行约束）

1. 一次任务只允许一个主模块。
2. 场景层禁止直接改核心规则状态。
3. 先写契约再写代码，契约变更需记录。
4. 每个任务必须附验证步骤与结果。
5. 跨模块改动默认为高风险（L2），先审批后实施。

---

## 7. 风险与回滚策略

主要风险：

1. 重构期行为漂移（看起来改了结构，实际玩法变了）。
2. 存档字段变更导致读档失败。
3. 双轨内容源导致线上行为不一致。

应对策略：

1. 每阶段设置可运行里程碑，不通过不进入下一阶段。
2. 关键模块先加最小回归测试再改代码。
3. 以基线标签作为回退锚点，任务级小提交。
