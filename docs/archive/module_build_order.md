# 开发顺序与架构建议（基于参考库）

## 参考库现状（2026-02-15）

- 卡牌：446 条（核心体量最大）
- 遗物：184 条
- 能力/状态：139 条
- 事件：57 条
- 敌人：51 条
- 药水：47 条
- 角色：4 条

结论：先做“结算正确性”再做“内容规模”，否则会在卡牌、遗物、状态叠加处反复返工。

## 分阶段模块顺序（基础版 -> 扩展版）

### Phase 1：`battle_loop` + `card_system`（当前应优先）
目标：形成稳定的战斗最小闭环。

最小验收：
- 回合流：出牌 -> 结束回合 -> 敌人行动 -> 新回合。
- 牌区流转：抽牌堆/手牌/弃牌堆/消耗堆。
- 目标系统：单体、全体、自身。
- 中文 UI 可用。

### Phase 2：`effect_engine` + `buff_system`
目标：把“卡牌文本”变为可扩展的规则执行链。

最小验收：
- 效果队列（顺序结算）。
- 常见状态：力量、敏捷、虚弱、易伤、中毒、脆弱。
- 回合时机触发：回合开始/结束、打牌后、受击后。

### Phase 3：`enemy_intent`
目标：敌人行为从脚本硬编码升级为“意图+规则约束”模型。

最小验收：
- 条件动作与权重动作并存。
- 不可连续动作规则。
- 进阶等级（ascension）参数开关。

### Phase 4：`run_meta` + `reward_economy` + `relic_potion`
目标：从“单战斗 demo”升级为“可连续推进的一局游戏”。

最小验收：
- 战后奖励（金币、三选一卡、遗物/药水）。
- 遗物触发链。
- 药水槽与使用时机。

### Phase 5：`map_event`
目标：构建局内推进骨架。

最小验收：
- 地图节点：普通战/精英/休息/事件/商店/Boss。
- 节点推进与分支选择。
- 事件系统最小子集（先做 10 个高频事件）。

### Phase 6：`save_seed_replay` + `content_pipeline`
目标：保证可复现、可迭代、可平衡。

最小验收：
- RunState 存档/读档。
- 固定种子可复现。
- 内容表导入（卡牌/遗物/敌人/事件）与版本迁移。

## 架构硬规则

1. UI 层不直接改规则状态，只发命令/展示状态。
2. 战斗规则必须可在“无 UI”模式下执行（便于自动化验证）。
3. 敌人 AI 只产出“意图”，真正结算走 effect_engine。
4. RunState 与 BattleState 分离，避免跨层污染。
5. 每个模块先做 MVP，再做扩展，不并行堆功能。

## 基础版里建议先做的内容子集

- 角色：先仅铁甲战士。
- 卡牌：20~30 张（打击、防御、力量、消耗、格挡流核心）。
- 敌人：Act1 普通+精英共 8~12 组。
- 遗物：30 件高频。
- 事件：10 件高频。

这套子集能覆盖绝大多数系统交互，不会因为内容量过大拖慢重构节奏。
