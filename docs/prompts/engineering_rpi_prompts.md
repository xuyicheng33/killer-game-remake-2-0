# AI 工程师提示词模板（RPI）

用途：把这份文件直接复制给 AI 工程师，要求其按阶段输出，避免一次性大杂烩。

## Prompt A：Research（只研究，不改代码）

```md
你现在是本仓库的研究员，只做调研，不改任何文件。

任务ID：<task-id>
目标：<一句话目标>
范围边界：
- 包含：<范围>
- 不包含：<范围>

请输出：
1. 关键文件列表（必须带行号）
2. 当前行为描述（客观事实）
3. 风险点（按高/中/低）
4. 未知项（需要我确认的问题）

约束：
- 禁止给实现代码
- 禁止改动文件
- 结论必须引用代码锚点
```

## Prompt B：Plan（生成可执行计划）

```md
基于已确认的研究结论，为任务 <task-id> 生成实施计划。

请输出：
1. 目标与验收标准
2. 改动白名单文件（精确路径）
3. 分步实施清单（每步写清改哪些文件、改什么）
4. 验证命令（至少包含 workflow-check）
5. 风险与回滚策略

约束：
- 不得超出白名单
- 不得把多个模块混成一步
- 计划应支持逐步审阅
```

## Prompt C：Implement（按步执行）

```md
按已批准计划执行任务 <task-id>，严格一步一汇报。

执行规则：
1. 每次只做一个步骤
2. 每步完成后输出：
   - 已改文件
   - 改动摘要
   - 验证命令与结果
3. 若发现计划错误，先停止并给“计划修订建议”，不要擅自扩 scope

硬约束：
- 仅允许修改白名单文件
- 保持现有功能行为等价（除非计划明确要求改行为）
```

## Prompt D：Self Review（工程师自检）

```md
对任务 <task-id> 做交付前自检，输出如下格式：

1. 变更文件清单（按模块分组）
2. 行为变化清单（若无则写“无行为变化”）
3. 风险清单（S0/S1/S2）
4. 验证结果：
   - make workflow-check TASK_ID=<task-id>
   - 任务相关功能验证
5. 是否建议提交（是/否 + 理由）
```

## Prompt E：Handoff（交接产物）

```md
请生成本任务交接内容，回填到：
- docs/tasks/<task-id>/handoff.md
- docs/tasks/<task-id>/verification.md

要求：
1. 写清主路径与边界用例
2. 写清已知风险与影响范围
3. 给出建议 commit message
```
