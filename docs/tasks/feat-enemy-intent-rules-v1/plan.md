# 任务计划

## 基本信息

- 任务 ID：`feat-enemy-intent-rules-v1`
- 任务级别：`L1`（预期；若发现需要调整跨模块接口或变更 `docs/contracts/battle_state.md`，立刻升级为 `L2` 并停在审批点）
- 主模块：`enemy_intent`
- 负责人：Codex
- 日期：2026-02-15

## 目标

将敌人“意图选择”规则化，使行为更接近原版并与意图展示一致：

- 引入“不可连续动作”约束（在存在其他可选动作时禁止重复上一次动作）。
- 条件动作优先于权重动作（先判定条件池，再走权重池）。
- 进阶参数（ascension）提供占位且可配置（不引入跨模块契约/接口变更）。
- 同步意图展示：UI 展示必须反映最终被规则选中的动作。

## 范围边界

- 包含：
  - `enemy_intent` 内部意图规则层与选择算法
  - 敌人 scene / 数据中对“动作定义”的最小补充（用于条件/权重/不可连续的表达）
  - 意图 UI 与规则结果的映射联动
- 不包含：
  - Phase B/C/D 内容
  - 新的战斗存档字段/回放/随机数体系调整
  - `BattleState` 契约字段变更（若必须改，则升级为 L2 并等待“批准”）

## 实施步骤

1. 盘点现有意图系统：动作定义格式、选择入口、UI 展示链路、以及“上一动作”信息存放位置。
2. 在 `modules/enemy_intent/**` 内实现规则层（不改跨模块接口）：
   - 候选动作分为条件池与权重池。
   - 先筛选条件池（满足条件的动作集合），若非空则从中选取（支持稳定优先级或权重）。
   - 否则走权重池按权重抽取。
   - 应用“不可连续动作”约束：若存在与上次不同且可选的动作，则排除上次动作。
3. 增加 ascension 占位配置：
   - 在 `enemy_intent` 内提供可注入/可编辑的 `ascension_level`（默认 0）。
   - 不新增 `BattleState` 字段、不要求 `battle_loop`/`run_meta` 传参改接口。
4. UI 联动：
   - 将规则最终选中的动作 ID/intent 映射到意图图标与文本，确保展示与执行一致。
5. 补齐任务文档：
   - `handoff.md`：改动摘要、文件清单、风险、建议 commit message。
   - `verification.md`：2 条主路径 + 1 条边界验证，包含可复验步骤。

- 风险：
  - 规则层与敌人现有数据格式不匹配导致 intent/UI 不一致或选择异常。
  - 触发条件表达不当导致条件动作永远不触发或过度触发。
- 回滚方式：
  - 仅回滚本任务提交（不涉及历史提交改写）。
  - 将规则层切换回旧路径（若实现中保留旧选择函数作为 fallback）。
